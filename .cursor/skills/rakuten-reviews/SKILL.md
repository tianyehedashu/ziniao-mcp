# 乐天 RMS 店铺评价（口碑 / VOC）

面向**顾客之声（Voice of Customer）**与店铺声誉管理：在已登录 RMS 的浏览器会话中，通过 ziniao 拉取楽天 **店铺评价一览**（含ショップレビュー与商品レビュー），支撑**客服优先级与话术**、供应链与履约整改、商详与包装迭代；**与广告报表分轨**，不以投放指标替代口碑解读。

**文档与工具**：主说明见本文；[`references/SCRIPTS.md`](references/SCRIPTS.md) 载 **`analyze_reviews_csv.py`** 的产出说明、参数与 PowerShell 示例；可执行脚本在 **`scripts/`**（与乐天广告 Skill 的 `scripts/` + `references/SCRIPTS.md` 组织方式对齐）。扩展细节仅从本文链入 SCRIPTS，不叠套多级引用。

## 前提

1. 已 `open-store` / `launch` 并登录 RMS。  
2. 评价类 preset 会导航至 RMS 对应页面；以 `ziniao site list` / `rakuten --help` 为准。

## 店铺与会话与归档

1. **会话与店铺一致**：拉数前确认当前 ziniao 会话即目标店；换店后请**重新进入 RMS** 再执行，避免串店污染口碑数据。  
2. **落盘路径（多店建议带店铺段）**：多店同仓时，`-o` 宜为 `exports/<店铺简写或 site>/reviews_<区间>.csv`（如 `reviews_last30.csv`、`reviews_2026-03-01_2026-03-31.csv`），与广告 `batch_*` 及其它店文件区分，便于**溯源与对账**。单店可用 `exports/reviews_*.csv`。  
3. **交付物而非日志**：`ziniao` 成功提示仅在 **②** 点到为止；面向业务方的正文以 **四块结构 + Markdown 表格** 为主，避免粘贴大段控制台或原始 CSV。

## 业务能力菜单（默认以选项呈现给用户）

### 执行与交付规范（必遵守）

1. **首次响应**（用户只说「乐天评价 / 拉评论 / 口碑怎么样」等模糊需求时）：先给出下方 **A～F 编号菜单**，每项一行：**编号 + 短标题 + 一句业务价值**，末尾写「请回复 **编号**（可多项，如 `A+C`），并说明**统计区间**（起止日期或最近 N 天）。」  
2. **用户已点选编号**：  
   - **路径**：用户未指定时，Agent 按上文归档约定**自拟**清晰文件名，并在 **②** 写全路径。  
   - **时间窗**：口语如「最近 7 / 30 天、本月、上月」按 **JST 自然日**解析；优先 **`-V last_days=N`（含当日）** 或 **`start_date` / `end_date`** 固定窗。仅当「大促那段」等语义不清，或**多店会话不明**时，再极简确认。  
   - **表述**：面向业务方输出**可决策的结论**（风险 SKU、重复痛点、是否需公关/补偿），避免罗列参数与长串命令。  
3. **用户已说清目标**（例如「导出上月全部评价做复盘」）：可跳过菜单，直接执行并按四块回报。  
4. **对用户的交付结构（评论场景默认形态）**：**默认**四块，定性（原声主题）与定量（条数、占比、密度）结合，避免整屏脚本输出：  
   - **① 观测窗口 + JST 口径**（一行）：闭合起止 **或** `last_days`（日本自然日、含当日）；说明本批数据覆盖 **店铺评价 / 商品评价** 或注明分析子集。  
   - **② 数据落点**：导出 CSV 及解析产物的**仓库相对或绝对路径**；多店路径中含店铺标识。解析后默认侧车：`reviews_*_report_zh.md`（中文汇总）、`reviews_*_lowscore_zh.md`（中低分**全部**正文，按楽天键分组）；可选 `reviews_*_summary.json`、`reviews_*_lowscore_zh.csv`（见 [`references/SCRIPTS.md`](references/SCRIPTS.md)）。  
   - **③ 结论**：多 SKU 对比、星级结构、主题频次等，**优先 Markdown 表格**；列名用业务语言并标明**分母**。**低分**默认指 **★1～3（中差评）**，**低分率 = ★1～3 条数 / 该键总评条数**（分母须与列定义一致）。**★1～2** 建议并列一列（可标为「其中差评/极端低分」），便于公关与品控单独盯；用户明确要求「只看 ★1～2」时，在 **①** 或表脚**改口径并全文统一**。表后 **2～4 句**写清：整体口碑轮廓、异常是否呈**重复模式**（同款多评同因）、是否与**大促滞后、物流高峰**等同向，避免「只有表、没有判断」。中低分集中款可各附 **一句原声摘要**（不贴全文）。  
   - **④ 行动**：**≤3 条**、责任可归属（客服 / 仓储 / 商详 / 品控）。用户点 **F** 或索要完整清单时，可扩至 **3～5 条**，默认仍以 **≤3 条** 控篇幅。  
5. **下钻**：仅在用户要求时展开：典型差评摘录、按周滚动、指定商品键下的完整线索；默认**不**贴长 CSV。  
6. **样本与义务**：量不足时在 **③** 标明**统计效力**（如总评过少则低分率波动大），并给出「延长窗口 / 次月复测」建议，不得用简短省略结论。

### 评论场景专用口径（Agent 内化）

- **ショップレビュー**：偏整体服务、发货、沟通；**商品レビュー**：偏单品质量、描述相符、使用体验。默认 **③** 中若合并统计须注明，**更推荐分述**以免「店铺情绪」掩盖「单品缺陷」。  
- **低分 / 中低分（默认口径）**：**★1～3**。楽天侧 ★3 常带建设性不满，纳入监测可减少「只看 1～2 星漏掉中评差评」的盲区。  
- **差评 / 极端低分（窄口径）**：**★1～2**，舆情与补偿优先级更高；与上条**同表并列**或按用户指定**二选一为唯一口径**时，须在 **① 或表脚一次性声明**，全文不得混用。  
- **主题 / 聚类**：从正文归纳**可行动标签**（物流破损、说明书错配、尺寸预期差、做工卡扣等），忌空泛「不好」；重复出现的具体描述优先于单次情绪宣泄。  
- **隐私**：下钻展示订单号、收货信息时**脱敏或概括**。

### Agent 默认自动化（减负）

1. **拉数主路径**：`ziniao rakuten reviews-csv -o <路径> -V last_days=N` 或 `-V start_date=` + `-V end_date=`；落盘编码以 CLI 提示为准（常见 **CP932 → UTF-8**）。  
2. **C～E 分析与中文侧车**：拉数后对**同一 CSV** 运行 **`scripts/analyze_reviews_csv.py`**（路径：`.cursor/skills/rakuten-reviews/scripts/analyze_reviews_csv.py`，用法见 [`references/SCRIPTS.md`](references/SCRIPTS.md)）。默认生成 **中文** `*_report_zh.md` 与 **中低分全文** `*_lowscore_zh.md`（含区间内**全部**低分条数，非抽样）；需要 Excel 或机读时再追加 `--lowscore-csv`、`--write-json`。对用户回复仍以 **四块 + Markdown 表格**为主，**勿**把脚本 STDOUT 或整份低分 MD 贴进正文；用户明确要求「逐条低分」时指引其打开 `*_lowscore_zh.md` / CSV。Windows 下**避免**多行 `python -c`；显式调用上述脚本路径。  
3. **商品聚合键**：与脚本一致，从 **商品レビュー** 的 `レビュー詳細URL` 解析 `…/item/1/<店ID>_<商品ID>/…`；与内部 SKU 映射由运营维护。  
4. **失败回退**：会话失效、缺少子命令时，按文末 **注意** 升级 ziniao / 重装并重试。

### 可选业务能力一览（对用户展示的选项文案）

| 编号 | 用户看到的名称（示例） | 能完成的具体业务 | 业务价值 |
|------|------------------------|------------------|----------|
| **A** | 活动 / 大促后口碑复盘 | 按**指定起止日期**导出该段全部评价记录 | 对齐活动投放与售后反馈，判断**是否伤店誉、是否需公关或补偿策略** |
| **B** | 滚动周期巡检 | 按 **`last_days`** 导出最近 N 个**日本时间自然日**（含当天） | **固定节奏监测**：中低分率（默认 ★1～3）、主题词是否突变，适合作周例会与预警 |
| **C** | 差评与商品优先级 | 按 **楽天商品键 / 内部 SKU（若有映射）** 聚合**中低分密度**（默认 ★1～3）与主题，可并列 ★1～2 | 将客服、仓储、商详资源投向**高影响商品**，避免平均用力 |
| **D** | 星级与「体感」对齐 | 同期 **星级分布、中低分与 ★1～2 占比** | 校验「体感差评变多」是**趋势还是噪声**，减少误判内耗 |
| **E** | 品控与履约线索挖掘 | 从正文提炼 **物流、品质、描述相符度** 等主题 | 为供应链与页面优化提供**可验证的假设**（哪类痛点增速最快） |
| **F** | 评价侧行动清单 | 在已有或刚拉取的数据上，按本 Skill「运营要点」节归纳 | 产出可执行项；**默认与交付块 ④ 合并为 ≤3 条**，展开时 **3～5 条** |

**组合建议**：大促后必做 **A**；日常 **B**；要给商品/采购开会带 **C+E**；向管理层一句话结论用 **D+F**。

### 编号与数据来源对应（Agent 执行用；对用户的回复里不必展开命令）

| 编号 | 主要手段 |
|------|----------|
| A | `ziniao rakuten reviews-csv`，`-V start_date=` + `-V end_date=`，`-o` 落盘 |
| B | 同上，`-V last_days=N`（与起止日期同时传入时，**起止日期优先**） |
| C～F | 基于 A 或 B 的 CSV；**C/E 优先**跑 `analyze_reviews_csv.py` 得中文报告与低分全文；F 另依赖下文「基于评价数据的运营要点」 |

### 用户可说的极简话术（仍触发选项）

- 「乐天评价帮我看看」→ Agent 先输出 **A～F 菜单**。  
- 「大促结束复盘口碑」→ **A**（若缺具体日期再确认）。  
- 「最近一周有没有差评炸」→ **B**，Agent 直接 `last_days=7` 并四块交付。

### ③ 结论的推荐表格（按需选用列）

| 分析类型 | 建议表头（业务语言） |
|----------|----------------------|
| C 商品优先级 | 楽天商品键 / 代表商品名(缩) / 商品评论数 / 中低分(★1～3)条数 / 中低分率 / 其中★1～2条数(可选) / 主要痛点主题 / 原声摘要(可选) |
| D 星级结构 | 星级(1～5) / 条数 / 占比(%) / 备注（店铺评论 / 商品评论分列或注明合并） |
| E 主题挖掘 | 主题标签 / 命中条数 / 时间说明（单窗写「本观测期」；多期再写环比） |

**店铺评论与商品评论**宜分表或分节，避免将「对店服务印象」与「对货体验」混为同一占比。

## 基于评价数据的运营要点

**可读性**：用户未索要深度解读时，**勿全文复述**本节；默认交付仍为 **四块 + 表格 + ≤3 行动**。点 **F**、追问根因或下钻时再引用下列原则。

1. **日界**：`last_days` 为 **JST 自然日**且**含当日**；境外团队勿将本地日期直接等同楽天日界。  
2. **窗口选择**：活动复盘用 **A（起止）**；日常监测用 **B（last_days）**；二者对照可区分「脉冲客诉」与「结构性问题」。  
3. **因果谨慎**：订单脉冲后评价往往**滞后**；单点暴增未必等于质量恶化，需结合活动与物流节奏解读。  
4. **处置优先级**：**同款、同因、多次出现**的具体描述优先于孤立谩骂；可映射到 SOP 的（破损、错说明书、安装困难）优先于不可控情绪。  
5. **低分口径**：默认以 **★1～3** 做率与密度；**E 主题挖掘**优先采信 **★1～3** 正文，五星仅作对照。用户指定窄口径时改为仅 ★1～2 并全局一致。  
6. **编码**：源导出常见 CP932；ziniao 侧常已 UTF-8。乱码时先核对用 UTF-8 打开。仓库文档路径以**实际存在文件**为准。

## 注意

- **CLI 与预设**：子命令来自内置 `ziniao_mcp/sites/rakuten/reviews-csv.json`。若 `ziniao site list` / `ziniao rakuten --help` 中**没有** `reviews-csv`，说明当前可执行文件对应的是**旧版包**；请在本仓库执行 **`pip install -e .`**（或安装与仓库一致的发行版）后重开终端，或用 **`python -m ziniao_mcp.cli`**。若使用 **`uv tool install`** 到 `~/.local/bin`，升级时 exe 被占用需先结束进程再 **`uv tool install . --force`**。  
- **PowerShell**：多条命令请**逐行执行**，或同一行用 **分号 `;`** 串联；**不要**使用 **`&&`**（Windows PowerShell 5.1 不支持；PowerShell 7+ 支持 `&&`，为兼容 5.1 文档统一用分号或换行）。
