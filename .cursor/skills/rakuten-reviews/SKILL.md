# 乐天 RMS 店铺评价

在已登录 RMS 的浏览器会话中，通过 ziniao 拉取**店铺评价一览**数据，用于口碑管理、品控与运营决策（非广告报表）。

## 前提

1. 已 `open-store` / `launch` 并登录 RMS。  
2. 评价相关 preset 会导航至 RMS 内对应页面；以 `ziniao site list` / `rakuten --help` 为准。

## 业务能力菜单（默认以选项呈现给用户）

### Agent 输出规范（必遵守）

1. **首次响应**（用户只说「乐天评价 / 拉评论 / 口碑怎么样」等模糊需求时）：先给出下方 **A～F 编号菜单**，每项一行：**编号 + 短标题 + 一句业务价值**，末尾写「请回复 **编号**（可多项，如 `A+C`），并说明**统计区间**（起止日期或最近 N 天）。」  
2. **用户已点选编号**：仅追问缺失信息（**日期**、滚动天数、落盘路径）；再执行拉数并输出**业务结论**（高危 SKU、差评主题、建议动作），避免堆砌技术细节。  
3. **用户已说清目标**（例如「导出上月全部评价做复盘」）：可跳过菜单，直接执行并回报结论。

### 可选业务能力一览（对用户展示的选项文案）

| 编号 | 用户看到的名称（示例） | 能完成的具体业务 | 业务价值 |
|------|------------------------|------------------|----------|
| **A** | 活动 / 大促后口碑复盘 | 按**指定起止日期**导出该段全部评价记录 | 对齐活动投放与售后反馈，判断**是否伤店誉、是否需公关或补偿策略** |
| **B** | 滚动周期巡检 | 按 **`last_days`** 导出最近 N 个**日本时间自然日**（含当天） | **固定节奏预警**：差评率、关键词是否突变，适合周会材料 |
| **C** | 差评与商品优先级 | 在导出结果上按 **SKU / 品番** 聚合差评密度与主题 | 把客服、仓储、商详整改资源集中到**真正拖口碑的款**，避免平均用力 |
| **D** | 星级与「体感」对齐 | 同期数据里看评分分布、极端差评占比 | 解释「觉得最近差评多」是**事实还是样本偏差**，减少无效内耗 |
| **E** | 品控与履约线索挖掘 | 从评价文本中归纳 **物流、品质、描述不符** 等主题 | 给供应链与页面改版**可验证的假设**（哪类抱怨上升最快） |
| **F** | 评价侧行动清单 | 在已有或刚拉取的数据上，按本 Skill「运营要点」节归纳 | 输出 **3～5 条可执行项**（先改哪几款、先回复哪类客诉），避免只有数据没有下一步 |

**组合建议**：大促后必做 **A**；日常 **B**；要给商品/采购开会带 **C+E**；向管理层一句话结论用 **D+F**。

### 编号与数据来源对应（Agent 执行用；对用户的回复里不必展开命令）

| 编号 | 主要手段 |
|------|----------|
| A | `ziniao rakuten reviews-csv`，`-V start_date=` + `-V end_date=`，`-o` 落盘 |
| B | 同上，`-V last_days=N`（与起止日期同时传入时，**起止日期优先**） |
| C～F | 基于 A 或 B 产出文件做表外分析；F 另依赖下文「基于评价数据的运营要点」 |

### 用户可说的极简话术（仍触发选项）

- 「乐天评价帮我看看」→ Agent 先甩 **A～F 菜单**。  
- 「大促结束复盘口碑」→ **A**（再问具体日期范围）。  
- 「最近一周有没有差评炸」→ **B**，`last_days=7`。

## 基于评价数据的运营要点

1. **日界**：`last_days` 按 **日本时间自然日**计，且**含当日**；跨时区团队勿用本地日历硬套。  
2. **区间**：需要绝对窗口用 **起止日期**；需要「含今天共 7 天」用 **`last_days=7`**。  
3. **解读**：单日暴增可能是活动引单后的**正常滞后**；应结合 **B 滚动**与 **A 活动窗**对照，避免过度反应。  
4. **动作优先级**：优先处理 **重复出现的具体缺陷描述**（同款多评提及），其次才是零星情绪化差评。  
5. **编码**：导出文件常见 **CP932**，下游分析前转 UTF-8；详见仓库 **`docs/site-fetch-and-presets.md`**（以当前仓库路径为准）。
