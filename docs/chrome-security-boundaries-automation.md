# Chromium 安全边界与自动化：和「反爬隐身」不是同一战场

本文说明：**页面侧反自动化（bot 指纹）** 与 **浏览器/Chromium 安全策略（可信用户事件、用户激活、权限与敏感 API）** 的差异；并列举常见 **CDP / WebDriver 难以等价真人** 的设计。实现细节会随 **Chromium 版本与站点代码** 变化，表格以「典型行为」表述；上线前请在目标环境实测。

---

## 1. 两个层次：不要混为一谈

| 层次 | 典型问题 | 常见手段 | 自动化能否「博弈」 |
|------|----------|----------|-------------------|
| **页面识别 bot** | `navigator.webdriver`、Canvas/WebGL、行为熵、TLS 指纹等 | 隐身补丁、拟人轨迹、住宅代理等 | **常能**缓解或绕过（与站点对抗强度有关） |
| **浏览器安全边界** | `Event.isTrusted`、自动填充提交语义、权限弹窗、支付/WebAuthn 等 | 产品设计 + 平台策略 | **多数不是**给纯 CDP 自动化留的后门；只能换路径或人机协同 |

**结论：** 隐身/拟人解决的大多是 **「页面 JS 能不能猜你是自动化」**；下列能力解决的是 **「浏览器愿不愿意把这次交互当真人安全语义」** —— 二者重叠有限。

---

## 2. 核心概念（严格区分）

### 2.1 `Event.isTrusted`

- **含义：** 事件是否由 **用户代理可信路径** 产生（例如真实键鼠由 OS 传入浏览器再派发到 DOM）。
- **CDP / WebDriver 典型情况：** `Input.dispatchMouseEvent`、`dispatchKeyEvent`、以及由脚本发起的 `element.click()` / `dispatchEvent()` 派生的事件，在页面里 **`isTrusted === false`** 是常态。
- **和隐身的区别：** 改 `navigator.webdriver` **不会**把合成点击变成 `isTrusted: true`。

### 2.2 用户激活（User activation / Transient activation）

- **含义：** HTML 规范中的 **瞬态用户激活**，用于门控 `window.open`、部分剪贴板、自动播放等能力。
- **CDP `Runtime.evaluate` 的 `userGesture: true`：** 让 **本次 `evaluate` 内执行的 JS** 在 Chromium 里 **带有用户手势上下文**（利于部分需激活的 API），**不等于** 为后续 **单独一次 CDP 合成鼠标事件** 打上「真人 DOM 事件」标签，也 **不保证** 与 **密码自动填充提交** 等内部逻辑等价。

### 2.3 小结对照

| 机制 | 解决的主要问题 | 能否替代「真人点一下再提交」 |
|------|------------------|------------------------------|
| 隐身 / 拟人 CDP 输入 | 页面指纹与行为统计 | **否**（不触及 isTrusted 语义） |
| `userGesture: true`（evaluate） | 部分 API 的激活门控 | **部分场景**；对 autofill 等 **勿默认假设可用** |
| OS 级真实键鼠 | 向窗口注入硬件级输入 | **最接近**真人（仍受焦点、权限、多显示器等约束） |
| 用户手动一步 | 真人可信路径 | **最可靠** |

## 3. 能力表：典型「严格 / 多方案」场景（Chromium 向）

下列均为 **常见设计意图**；具体以 **当前 Chromium + 目标站点** 为准。

| 领域 | 浏览器/平台在防什么 | CDP / 脚本自动化典型局限 | 可行替代思路（多方案） |
|------|---------------------|---------------------------|-------------------------|
| **DOM 可信事件** | 脚本冒充用户点击/键入 | 合成事件 **`isTrusted === false`** | OS 级输入；或接受 **人工点击/键入关键一步** |
| **密码自动填充（Autofill）** | 未提交的预览值随表单「误提交」、凭据泄露 | 合成点击可能 **无法触发** 与真人相同的 **提交/填充提交** 语义 | **自备密码** 写入控件（`fill`/`type`）再提交；或 **人工**完成登录再继续自动化 |
| **剪贴板读取** `readText()` | 恶意页面偷剪贴板 | 常需 **安全上下文 + 权限 + 用户激活**；纯自动化易失败 | `evaluate` + `userGesture: true` **或** 用户授权一次；或改流程不读剪贴板 |
| **剪贴板写入** | 静默写剪贴板骚扰用户 | 策略因上下文而异 | 同上；或仅用 CDP/扩展侧能力（若有） |
| **`window.open` 弹窗** | 无用户交互乱开窗口 | **无用户激活** 常被拦截 | `evaluate` + `userGesture: true` 打开；或改为同页 `navigate` |
| **通知 / 地理位置等权限** | 静默索取敏感能力 | 自动化下 **弹窗与策略** 与真人流程不一致 | 预先站点设置授权、策略白名单；或 **人工点允许** |
| **媒体自动播放（带声）** | 自动出声骚扰 | 无用户激活时 **常静音或拒绝** | 用户手势后播放；`evaluate`+`userGesture` **在测试中有用**，生产依赖策略 |
| **全屏 / 画中画等** | 全屏欺骗、覆盖攻击 | 多需 **可信手势或明确用户意图** | 人工确认；或降级为非全屏流程 |
| **`getDisplayMedia`（屏幕共享）** | 窃取屏幕 | **强依赖用户选择与手势** | 几乎必须 **人工**选源并确认 |
| **File System Access（文件选择器）** | 任意读盘 | **`showOpenFilePicker` 等** 典型需 **用户手势** 打开的 UX | CDP **`DOM.setFileInputFiles`** 等路径上传（仅限 `<input type=file>`）；或人工选文件 |
| **Payment Request / SPC 等** | 未经确认的支付 | **强用户校验与厂商流程** | **不可**指望纯 CDP 绕过；改 API 集成或人工支付 |
| **WebAuthn / Passkeys** | 无硬件证明的认证 | 需 **认证器与用户在场证明**（密码学） | 生产：**安全密钥/平台认证器 + 人工**；自动化测试：Chromium **虚拟认证器（Virtual Authenticator）** 等 **测试专用** 路径 |
| **跨站 Cookie / 存储分区** | 跟踪与侧信道 | 非「真人」问题，但 **自动化脚本若依赖跨站共享** 会表现为「做不到」 | 按 **CHIPS/分区存储** 重新设计登录与域；同站请求 |
| **第三方 Cookie 淘汰** | 跨站跟踪 | 嵌入登录态丢失等 | **第一方登录**、Storage Access API（仍受策略与手势约束） |
| **证书 / 安全网络** | MITM、明文密码 | 错误证书、混合内容被拦 | 修正证书链与 **HTTPS**；勿依赖降级不安全模式 |

## 4. 与 ziniao 实践的对应关系（便于内部对齐）

- **隐身三层**（协议、JS 补丁、拟人）：主要覆盖 **§3 表格左列「页面识别 bot」** 相关能力，**不**改变 Chromium 对 **DOM `isTrusted`** 与 **自动填充内部语义** 的判定。
- **`Runtime.evaluate` + `user_gesture: true`**（如 `ziniao_mcp/core/_eval.py` 中 `safe_eval_js`）：服务于 **需用户激活的 JS API** 的评估路径，**不应**理解为「全局真人模式」。
- **RPA 落地**：对 **登录 / 支付 / WebAuthn / 系统文件选择** 等行，在方案里预留 **「检测到即暂停 → 人工一步 → 再继续」** 比强行纯 CDP 更稳。

## 5. 维护说明

- Chromium 行为以官方发行说明与安全策略为准；站点还可叠加 **自定义 `isTrusted` 校验**（合法用途包括防机器人与防欺诈）。
- 若本文与 `docs/stealth.md` 并列阅读：**stealth.md** 偏实现与补丁清单；**本文**偏 **平台安全边界与预期管理**。
